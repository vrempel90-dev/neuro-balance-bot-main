# Neuro Balance — AI-бот записи на приём. Пакет для передачи разработчику

Этот архив — выжимка из работающего CRM Neuro Balance: всё, что нужно, чтобы
переписать/поднять бота записи пациентов в WhatsApp. Бот сейчас в проде, работает
на **OpenAI `gpt-4o-mini`** с нативным function-calling, общается через **Wazzup**.

> Все тексты (промпт, шаблоны ответов, цены, адрес) — **источник истины**.
> Их писали и выверяли операторы клиники. Не переформулируй — переноси дословно.

## 1. Что внутри архива

  bot-clinic-info.ts      ← 18 дословных шаблонов ответов операторов (get_clinic_info)
  bot-ai.ts               ← вызов LLM (OpenAI + Anthropic), tool-calls маппинг
  openai.ts               ← клиент OpenAI + Whisper (транскрипция голосовых)
  phone-validation.ts     ← нормализация казахстанских номеров

src/app/api/bot/

  check-slots/route.ts    ← GET  /api/bot/check-slots — свободные слоты (HTTP-контракт)

> 💡 Если читать TypeScript-сборщик `bot-prompt.ts` неудобно — открой
> `prompt/SYSTEM_PROMPT_rendered.md`: там весь промпт целиком, с подставленными
> фактами клиники, ровно как его видит модель.

Чего тут **нет** (это инфраструктура текущего CRM, при переписывании бота она
своя): движок диалога `bot-engine.ts`, входящий вебхук Wazzup, слой БД
`crm-queries.ts`/`chat-queries.ts`, Telegram-уведомления, авто-кампании. Если
понадобятся — запросите отдельно, но для нового бота важна логика и контракты
ниже, а не наша обвязка.

## 2. Архитектура в двух словах

```
Пациент (WhatsApp)
   │
   ▼
Wazzup webhook ──► наш бэкенд ──► buildSystemPrompt() + история диалога
                                        │
                                        ▼
                              OpenAI gpt-4o-mini (tool-calling, temp 0.7, max_tokens 1000)
                                        │  модель вызывает инструменты
                                        ▼
                              executeToolCall() ──► БД (слоты, запись, логи)
                                        │
                                        ▼
                              ответ пациенту обратно через Wazzup
```

Цикл: получили сообщение → собрали промпт → дернули LLM → если модель вызвала
инструмент, выполнили его и отдали результат обратно в LLM (tool-loop) → когда
модель вернула текст, отправили пациенту. Все вызовы инструментов логируются в
таблицу `bot_action_logs` — на них же завязаны программные «гейты» (см. §5).

## 3. Системный промпт (`bot-prompt.ts`)

Промпт собирается динамически на каждое сообщение функцией `buildSystemPrompt()`.
Секции (в порядке сборки):

1. **Идентичность** — роль («администратор клиники Neuro Balance»), берётся из
   конфига `bot_config.system_prompt`. Должна быть 3–4 предложениями, НЕ полным
   промптом.
2. **Язык** — ru/kz.
3. **Профильные / непрофильные заболевания** — из конфига.
4. **Врачи клиники** — из БД (специализации).
5. **Текущая дата/время** — Астана UTC+5.
6. **Факты о клинике** — ⚠️ ЗАХАРДКОЖЕНЫ в промпте (адрес, цены, график, методы,
   прейскурант, рассрочка, Отдел забот). Без этой секции модель выдумывала адреса
   и регалии. Менять адрес/цену надо **одновременно** здесь И в `bot-clinic-info.ts`.
7. **Тон и стиль** — фиксированная палитра эмодзи, лексика («окошко», не «слот»),
   правила эмпатии, обработка голосовых, пациентов не из Астаны.
8. **Кто работает в ближайшие дни** — снапшот расписания (сегодня/завтра) с
   реальными свободными слотами. Сужает выбор, чтобы модель не предлагала врача,
   которого нет в графике.
9. **Сценарий диалога** — главный блок. 7 шагов (0–6):
   `проверка существующей записи → имя → жалоба → возраст → удобное время →
   слоты → чек-лист противопоказаний → запись`. Плюс общие правила (не дублировать
   авто-кампанию, дробление сообщений через `---`, не выдумывать врачей).
10. **Возражения** — каталог отказов и как их отрабатывать (один re-engage, потом
    отпустить).
11. **Эскалация** — кейсы, где бот не закрывает сам (повторный пациент, претензия,
    запись на третье лицо, неподвижный пациент).
12. **Few-shot примеры** — образцы тона и дробления.
13. **Чек-лист противопоказаний** — единственный текст, который бот зачитывает
    пациенту дословно.

Полный финальный текст промпта можно сгенерировать, вызвав `buildSystemPrompt()`
с реальным конфигом/врачами/расписанием — структуру и все статические секции
видно прямо в `bot-prompt.ts`.

## 4. Инструменты бота (`bot-tools.ts` → `BOT_TOOLS`)

9 функций (OpenAI function-calling). Порядок вызова задан сценарием:

| Инструмент | Назначение |
|---|---|
| `find_existing_appointment` | ВСЕГДА первым: есть ли у пациента активная запись (по телефону) |
| `record_chief_complaint` | зафиксировать жалобу + профильность. **Гейт** для записи |
| `get_clinic_info` | дословный шаблон ответа на типовой вопрос (18 топиков) |
| `get_doctor_info` | подобрать врача под диагноз (только из БД, не выдумывать) |
| `check_available_slots` | свободные слоты на дату |
| `verify_contraindications_check` | вердикт по чек-листу. **Гейт** для записи |
| `book_appointment` | создать запись (телефон берётся из WhatsApp, не передаётся) |
| `escalate_to_human` | передать оператору |
| `mark_irrelevant` | непрофильная заявка |

Схемы и полные описания (важны для качества!) — в `BOT_TOOLS`. Маппинг в формат
OpenAI/Anthropic — в `bot-ai.ts` (`toOpenAITools` / `toAnthropicTools`).

## 5. Программные «гейты» — почему промпта недостаточно

`handleBookAppointment` НЕ доверяет модели на слово. Перед созданием записи
проверяет по `bot_action_logs`, что в этом диалоге реально были:

1. `record_chief_complaint` с **профильной** жалобой (маркер `COMPLAINT_OK`);
2. `verify_contraindications_check` с вердиктом **proceed** (маркер `verdict=proceed`);
3. **НЕ** было вердикта `refuse` (иначе модель могла «уломать» пациента и пройти
   проверку повторно — это заблокировано).

Если гейт не пройден — инструмент возвращает модели строку-инструкцию (не запись),
и диалог продолжается. Это медицински/юридически важная отсечка — сохраните её
при переписывании.

Дополнительно: если LLM прислала несуществующего врача или врача вне графика —
запись не теряется, а уходит в **«Резерв»** (force-запись) с пометкой для
администратора + Telegram-уведомлением. См. fallback-логику в `handleBookAppointment`.

## 6. Противопоказания (`bot-contraindications.ts`)

Утверждено клиникой 2026-05-07. Чистая функция `evaluateContraindications()`:

- **Возраст вне [16, 74]** → `refuse` (принимаем до 75, т.е. 75 уже нет).
- **Hard-stop** (кардиостимулятор/имплант, беременность, активное онко, эпилепсия)
  → `refuse`.
- **Soft-stop** (тромбоз/кровотечение, декомпенсированный СД/тиреотоксикоз, острая
  инфекция/лихорадка, тяжёлая ССН/ДН, психоз в обострении) **или** любой нечёткий
  ответ → `escalate`.
- Иначе → `proceed`.
- Если у пациента уже есть подтверждённая запись (`appointmentLinked`) — любой
  `refuse` превращается в `escalate` (не теряем одобренного оператором клиента).

Текст чек-листа, который зачитывают пациенту — в `bot-prompt.ts`, секция
«Чек-лист противопоказаний».

## 7. HTTP-контракт записи (для внешнего бота)

Если новый бот будет жить отдельным сервисом и ходить в наш CRM по HTTP — есть
два готовых эндпоинта. Оба защищены заголовком `x-bot-secret: $BOT_API_SECRET`.

### Свободные слоты
```
GET /api/bot/check-slots?date=YYYY-MM-DD&doctor=<login?>
x-bot-secret: <BOT_API_SECRET>

→ 200 {
  "availability": [
    { "doctorLogin": "zhuma_md", "doctorName": "...", "date": "2026-06-02",
      "availableSlots": ["09:00","09:40","14:00"] }
  ]
}
```
`doctorLogin: "reserve"` = свободное время без привязки к врачу. Пациенту слово
«Резерв» НЕ говорить — «примет первый освободившийся врач».

### Создать запись
```
POST /api/bot/book
x-bot-secret: <BOT_API_SECRET>
Content-Type: application/json

{
  "patientName": "Иван Иванов",      // обязательно
  "phone": "77011234567",            // обязательно (11 цифр, 77...)
  "doctorLogin": "zhuma_md",         // обязательно
  "doctorName": "Жумабек ...",       // опц. (для отображения)
  "date": "2026-06-02",              // обязательно YYYY-MM-DD
  "timeStart": "14:00",              // обязательно HH:MM
  "notes": "45 лет, боли в пояснице",// опц.
  "conversationId": 123,             // опц. (для статуса диалога)
  "leadId": 456                      // опц.
}

→ 201 { "appointmentId", "doctorName", "date", "timeStart", "timeEnd" }
→ 400 если не хватает обязательных полей
→ 401 если неверный x-bot-secret
→ 500 { "error": "<текст>" }  // напр. слот занят
```

Под капотом оба зовут `createAppointment()` / `getAvailableSlots()` из
`crm-queries.ts`. Сигнатура `createAppointment`:
```ts
createAppointment({
  leadId?: number | null
  doctorLogin: string
  doctorName: string
  patientName: string
  phone: string
  date: string        // YYYY-MM-DD
  timeStart: string   // HH:MM
  notes?: string
  createdBy: string   // 'bot'
  force?: boolean     // обойти валидацию расписания (для Резерва)
}): Promise<Appointment>
```
Бросает `SlotConflictError` (слот занят) и `DoctorNotScheduledError`
(врач вне графика: `no_schedule | day_off | outside_hours | lunch_overlap`) —
их нужно ловить и делать fallback в Резерв (см. §5).

> ⚠️ HTTP-эндпоинт `/api/bot/book` — упрощённый: он НЕ проверяет гейты жалобы и
> противопоказаний (их обеспечивает встроенный `handleBookAppointment`). Если
> внешний бот ходит по HTTP — все проверки противопоказаний/жалобы он обязан
> делать у себя ДО вызова `/api/bot/book`.

## 8. Телефоны (`phone-validation.ts`)

Канон KZ-номера — `77XXXXXXXXX` (11 цифр). `sanitizeKzPhone()` приводит
`87...`/`+7 7...`/`7XXXXXXXXX` к канону. В Wazzup телефон контакта — 10 цифр без
кода страны. Телефон пациента бот НЕ спрашивает — он известен из WhatsApp.

## 9. Модель и параметры LLM (`bot-ai.ts`, `openai.ts`)

- Провайдер по умолчанию: **OpenAI**, модель **`gpt-4o-mini`**.
- `temperature: 0.7`, `max_tokens: 1000`.
- Голосовые сообщения транскрибируются **Whisper-1** (`transcribeVoice` в
  `openai.ts`), дальше обрабатываются как текст.
- В коде есть и ветка **Anthropic** (`claude-sonnet-4`) — переключается полем
  `provider`. Сейчас в проде не используется (нет ключа), но готова.
